# PRI-1517 — Truthful action results + tightened screenshot timeout

**Status:** draft
**Date:** 2026-05-06
**Linear:** [PRI-1517](https://linear.app/prime-radiant/issue/PRI-1517)
**Author:** Wash (Bob bc258b31/Opus 4.7)
**Considered alternative:** `2026-05-06-pri-1517-nav-observer-considered-and-rejected.md`

## Problem

A bundled `click(button:contains('Post'), return_screenshot:true)` against a trivial Bun.serve form-post fixture takes 30,003 ms and returns:

```
Error: CDP command timeout: Page.captureScreenshot
```

Repro: `examples/tutorial/.gauntlet/results/tutorial-06-post-and-verify_20260505T233203Z_s2pr/run.jsonl` events 37→38.

Two independent bugs present together:

1. **Action-result coupling.** `adapter.ts:884-903` chains `...await takeReturnScreenshot()` into the click result. A screenshot timeout becomes the click's tool result text: `Error: CDP command timeout: ...`. The click itself succeeded — the post was created — but the agent reads "click failed."

2. **Excessive timeout.** When the post-action screenshot fails to return promptly, the pooled client waits the full 30,000 ms default (`chrome-ws-lib.js:221-224`) before timing out. Most of that wall-time is wasted: the agent's next call (a fresh screenshot 3s later) succeeds in 60ms. The proximate cause is the run.jsonl evidence shows the pool *did* time out (the stderr `Pooled connection failed, using single-use:` warning fired) and then the agent harness's own `toolTimeoutMs:30000` killed the call shortly after. The deeper cause — what made Chrome (or the pool) take 30s to respond instead of 60ms — is not pinned down here. Plausible candidates include Chrome silently dropping requests across renderer-replacement during navigation, pool-internal request-id loss adjacent to PRI-1446, or eventHandler-slot contention. This change makes the symptoms cheap and honest regardless of which is root cause; instrumentation is added (see below) so we can tune the cap from real data rather than from a story. The deeper cause may warrant its own ticket if the instrumentation reveals a frequency that justifies it — see the considered-and-rejected design for one possible shape of that fix.

## Goals

- A failed bundled screenshot never lies about whether the action succeeded.
- A failed bundled screenshot costs at most a few seconds of wall-time, not 30s.
- No new subsystems, no new sockets, no per-action overhead in the no-bug case.

## Non-goals

- Eliminating the underlying CDP navigation race (kept as the rejected design's goal; revisit if frequency justifies).
- Modifying `chrome-ws-lib.js`.
- Tool name / shape / arguments contract for any web tool.

## Decision summary

- **Decouple action result from screenshot result.** `takeReturnScreenshot` wraps the screenshot call in try/catch and never throws. On failure it returns a `screenshotSkipped` reason. Action handlers compose `text + (screenshot unavailable: <reason>)` when applicable; the action's primary text stays truthful regardless.
- **Tighten screenshot timeout for the `return_screenshot` path to 5s.** Explicit `screenshot` tool calls keep the existing 30s CDP default.
- **No nav awareness.** We accept that bundled screenshots can occasionally fail when racing a navigation. The agent recovers cleanly: it reads a truthful action result + skip note, takes a separate screenshot on the next turn (which works), and moves on.

## Design

### `src/adapters/web/adapter.ts` changes

**`takeReturnScreenshot` rewrite.** Currently (`adapter.ts:849-857`):

```ts
const takeReturnScreenshot = async (): Promise<{ image?: ToolResult["image"]; imagePath?: string }> => {
  if (!args.return_screenshot) return {};
  const tmpFile = join(tmpdir(), `gauntlet-screenshot-${Date.now()}.png`);
  await this.chrome.screenshot(tab, tmpFile, null, false);
  // ... read, base64, return image+imagePath ...
};
```

Becomes:

```ts
const takeReturnScreenshot = async (): Promise<{
  image?: ToolResult["image"];
  imagePath?: string;
  screenshotSkipped?: string;
}> => {
  if (!args.return_screenshot) return {};
  const tmpFile = join(tmpdir(), `gauntlet-screenshot-${Date.now()}.png`);
  try {
    await this.chrome.screenshot(tab, tmpFile, null, false, { timeoutMs: 5000 });
    const data = readFileSync(tmpFile);
    const imagePath = logger.saveScreenshot(Buffer.from(data));
    try { unlinkSync(tmpFile); } catch {}
    return { image: { data: Buffer.from(data).toString("base64"), mediaType: "image/png" }, imagePath };
  } catch (err) {
    try { unlinkSync(tmpFile); } catch {}
    const reason = err instanceof Error ? err.message : String(err);
    console.warn(`[gauntlet] return_screenshot skipped: ${reason}`);
    return { screenshotSkipped: reason };
  }
};
```

The 5000ms cap is the only constant introduced. It lives at the call site (passed as `timeoutMs`), not as a hidden default — explicit screenshot tool calls remain unaffected.

**Result-text composition.** All call sites that today spread `...await takeReturnScreenshot()` into a result need to compose the skip note when present. The count is ~16 *handler cases*, but several handlers (e.g., `click` at adapter.ts:884-903) call `takeReturnScreenshot` in **both** the success and error branches — both branches must migrate to `composeResult`. A helper:

```ts
function composeResult(text: string, screenshotResult: ScreenshotResult): ToolResult {
  if (screenshotResult.screenshotSkipped) {
    return { text: `${text} (screenshot unavailable: ${screenshotResult.screenshotSkipped})` };
  }
  return { text, image: screenshotResult.image, imagePath: screenshotResult.imagePath };
}
```

Call sites change from:

```ts
return { text: `clicked ${args.selector}${note}`, ...await takeReturnScreenshot() };
```

To:

```ts
return composeResult(`clicked ${args.selector}${note}`, await takeReturnScreenshot());
```

Mechanical, exhaustive, easy to review.

### `src/adapters/web/lib/chrome-ws-lib.js` — surgical addition

`screenshot()` (`chrome-ws-lib.js:1788`) currently takes `(tabIndexOrWsUrl, filename, selector = null, fullPage = false)` — no timeout opt — and the internal `sendCdpCommand` calls inherit the 30000ms default. We add a single optional fifth parameter `opts = {}` that threads `opts.timeoutMs` to the `Page.captureScreenshot` `sendCdpCommand` call. Does not touch the pool, does not change behavior for callers that pass nothing.

```js
async function screenshot(tabIndexOrWsUrl, filename, selector = null, fullPage = false, opts = {}) {
  // ... existing logic ...
  const result = await sendCdpCommand(wsUrl, 'Page.captureScreenshot', {
    format: 'png',
    fromSurface: true,
    captureBeyondViewport: fullPage,
    clip
  }, opts.timeoutMs);  // NEW: threads optional timeout; undefined → 30000ms default
  // ... rest unchanged ...
}
```

A new optional fifth parameter with a default doesn't change call sites; all upstream callers stay green. The change still touches an upstream-tracked file, so it gets a brief `// PRI-1517: optional timeoutMs` comment and a note in the upstream-sync doc. Acceptable cost.

**Alternatives considered:**

- *`Promise.race` in `takeReturnScreenshot` against a 5s timer.* Rejected — abandoning a pooled-WS request leaves the request id orphaned in `pendingRequests` until the eventual 30s pool timeout, which leaks memory and may interfere with subsequent calls if the response does eventually arrive on a recycled id.
- *Bypass `screenshot()` entirely and call `sendCdpCommandPooled` directly with a 5s timeout, since `sendCdpCommandPooled(wsUrl, method, params, timeout=30000)` already exposes the parameter.* Rejected — would require duplicating the ~40 lines of clip-computation/viewport-detection logic that `screenshot()` does. The 5-LOC opts-pass-through stays cleaner; the upstream-sync friction is one optional parameter, marked.

## Failing case, post-fix

```
1. Adapter receives click(button:contains('Post'), return_screenshot:true).
2. Calls chrome.click(...) — succeeds in ~50ms.
3. Calls takeReturnScreenshot:
   ├── Calls chrome.screenshot(..., { timeoutMs: 5000 }).
   ├── Pooled Page.captureScreenshot races the navigation. Chrome eats it.
   ├── 5s later, sendCdpCommand throws: 'CDP command timeout: Page.captureScreenshot'.
   ├── try/catch catches. Logs warn. Returns { screenshotSkipped: 'CDP command timeout: Page.captureScreenshot' }.
4. composeResult composes:
   "clicked button:contains('Post') (screenshot unavailable: CDP command timeout: Page.captureScreenshot)"
5. Tool returns. Total wall-time: ~5s. Action result truthful. No image.
6. Agent reads result. On next turn, takes a separate screenshot (60ms, works).
```

vs. today: 30s wall-time, `Error: CDP command timeout: Page.captureScreenshot` as the *entire* tool result text, agent uncertain whether the click happened.

## Testing

### T1 — `takeReturnScreenshot` unit tests

`test/adapters/web/adapter.test.ts` additions. Mock the `chrome` interface to throw on `screenshot`. Verify:

- `takeReturnScreenshot` returns `{ screenshotSkipped: <reason> }`, does not throw.
- `composeResult` produces `text + (screenshot unavailable: <reason>)` when skipped.
- `composeResult` produces `{ text, image, imagePath }` on success.
- Action's primary text is preserved verbatim regardless of screenshot outcome.

### T2a — Deterministic timeout-cap regression test

`test/adapters/web/adapter.test.ts` addition. Inject a `chrome.screenshot` mock that returns a never-resolving promise (or rejects after >5s with the CDP timeout error). Drive a `click(...)` with `return_screenshot:true`.

Assertions:
- Tool result returns within ~5.5s wall-time (5s cap + small headroom). Hard upper bound.
- Tool result text is *exactly* `clicked <selector> (screenshot unavailable: <reason>)`. No `Error:` prefix, no other framing.
- `image` and `imagePath` are both unset.

This is the deterministic regression gate. Catches: (a) the timeout cap silently growing back to 30s, (b) the compose-result helper malforming the skip-note text, (c) the action-result poisoning bug regressing.

### T2b — End-to-end nondeterministic test

`test/e2e/web-form-post-nav.test.ts`. Bun.serve fixture: single button that POSTs and 303-redirects. Scripted client: one `click(button, return_screenshot:true)`.

Assertions:
- Tool result returns within ~6s wall-time.
- Tool result text *starts with* `clicked button` (action's primary success text preserved).
- Either `image` is present (screenshot won the race) **or** text includes `(screenshot unavailable:` (screenshot lost the race).

This is a smoke test for the real path, not a strict regression gate — the navigation race fires nondeterministically and may not trigger on every CI run. T2a above is the deterministic complement.

### T3 — Tutorial repro re-run (manual sanity)

Re-run `tutorial-06-post-and-verify` post-fix. Confirm the click action drops from 30s to ≤5s wall-time. Confirm action result text is truthful regardless of screenshot outcome.

## Instrumentation

The `screenshotSkipped` warn log includes: tool name, elapsed ms, error message. One line per skip. This is the load-bearing data source for tuning the 5s cap from real-world targets — without it, we'd be picking the next number from another story.

A second one-shot log (`[gauntlet] return_screenshot tightened timeout active: 5000ms`) fires once per `WebAdapter.start()` so it's discoverable in any run that uses this code path.

## Risks

- **5s might be too aggressive for slow stage/prod targets.** If real-world apps need >5s for a legitimate screenshot, we'll see `screenshotSkipped` notes more often than expected. Mitigation: the warn-level log at every skip is the primary data source. Tune the constant up if seen, or make it per-target configurable in a follow-up.
- **The 5s cap is heuristic, not derived.** The failure-model uncertainty (see Problem §2) means we don't know exactly why the call hangs. 5s is "much smaller than 30s, big enough that legitimate screenshots succeed in practice." It is not derived from a measured distribution. Acceptable as a first cut paired with the instrumentation above.
- **Mechanical call-site updates risk a typo.** Mitigation: T1 unit tests cover the helper; T2 e2e covers the click path; rg/grep covers the spread → composeResult migration. ~16 sites, all in one file, all the same shape.
- **Upstream-sync friction from the chrome-ws-lib edit.** One added optional parameter with a default. The next sync may need a 1-line hand-port. Cost is real but small, and concentrated.
