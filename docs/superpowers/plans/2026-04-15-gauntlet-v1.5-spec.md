# Gauntlet v1.5 — Specification

> **Audience.** A future Bob picking this up cold. You have never seen the
> conclave, you have not read the v1 profiles spec, and you should not have
> to. Everything load-bearing is in this document.
>
> **Relationship to v1.** `docs/profiles-spec.md` is superseded by this
> file. Ponder's implementation notes
> (`docs/superpowers/specs/2026-04-14-profiles-implementation-notes.md`)
> remain the canonical record of the *CDP WebAuthn gotchas* and the
> *observer session*; those two pieces are explicitly preserved. Everything
> else in v1 — `profiles/`, `read_profile`, typed profile catalog,
> `format/profile.ts`'s dual duty — is being reshaped.

---

## 1. One-paragraph summary

Gauntlet v1.5 reframes "profiles" as "context": a freeform directory of
files the agent reads on demand via a generic `read` primitive, the same
way any coding-style agent works. `profiles/` becomes `.gauntlet/context/`;
`read_profile` is deleted in favor of `read`; the agent discovers context
via a compact directory tree injected into the system prompt at turn 0,
not via a typed catalog and not via a runtime list/search tool. Stories
run in isolated
browser state with no ordering assumptions; identity is expressed with the
"As X" convention in card bodies and the agent logs in on demand. Passkey
installation is preserved, decoupled from profile parsing, moved into the
web adapter, and takes an explicit context-file path. Ponder's observer
session plumbing stays untouched. The model floor is raised to Sonnet 4.6.
v1 is refactored in place — no Theseus rewrite.

---

## 2. Data model — `.gauntlet/context/`

### 2.1 Principle

`.gauntlet/context/` is **a data store, not a typed thing.** It is a
directory on disk. The agent sees whatever is in it. Gauntlet does not
parse, validate, or interpret filenames beyond showing them in the tree
listing. Authors put whatever they want in it — markdown profiles, JSON
passkey credentials, fixture files, notes, config snippets — and shape the
tree however helps the agent.

This is a deliberate inversion of v1's shape, which tried to teach the
runner about profiles as a category. The runner now knows nothing. The
agent is left to read comprehension and filesystem primitives, same as it
would be in a normal coding workflow.

### 2.2 Layout and path

- The directory is `<dataDir>/.gauntlet/context/`, where `dataDir` is the
  existing `AppConfig.dataDir` value (from `GAUNTLET_DATA_DIR` / `--data-dir`
  / the server's data-root, etc — unchanged).
- The leading `.gauntlet/` segment groups Gauntlet-owned metadata under one
  hidden directory, alongside the existing runtime layout. `results/`,
  `stories/`, etc. are *not* being moved under `.gauntlet/` in v1.5 —
  only context.
- Contents are user-owned. Gauntlet reads but never writes
  `.gauntlet/context/`. The runner must never create or modify files
  inside it.

### 2.3 File conventions (not schema)

There is no schema. These are conventions authors will probably adopt, and
all are optional. The agent's sole guide is the tree listing plus its own
reading comprehension of whatever files it opens.

**Recommended convention:** use directory names that match the agent-facing
identity and put an `identity.md` (or `alice.md`) alongside any structured
data files (credentials, passkeys, fixtures).

```
.gauntlet/context/
  alice/
    identity.md           # prose: who Alice is, what she cares about
    credentials.md        # "Username: alice@example.com / Password: ..."
    passkey.yaml          # CDP virtual-authenticator credential (optional)
    cookies.yaml          # Pre-loaded session cookies (optional)
  bob/
    identity.md
    credentials.md
  README.md               # optional: project-level notes for the agent
```

But equally valid:

```
.gauntlet/context/
  alice.md                # everything about Alice in one file
  bob.md
```

Or:

```
.gauntlet/context/
  users/alice/passkey.yaml
  users/bob/passkey.yaml
  notes/login-flow.md
```

The runner does not care. The system-prompt tree listing reflects whatever
is on disk; the agent uses `read` to pull content when it needs it.

### 2.4 Example — the anchor

Throughout this spec the Alice-and-Bob privacy example uses this layout:

```
my-project/
  stories/
    privacy-001.md
  .gauntlet/
    context/
      alice/
        identity.md
        credentials.md
        passkey.yaml
      bob/
        identity.md
        credentials.md
```

`alice/identity.md`:

```markdown
A privacy-conscious blogger with a small friend list. Writes 2–3 posts a
week about cooking. Careful about what she shares publicly.
```

`alice/credentials.md`:

```markdown
- Username: alice@example.com
- Password: hunter2
```

`alice/passkey.yaml` is whatever Ponder's `readPasskey` accepted in v1
(credentialId, rpId, privateKey, signCount, optional userHandle). The
parser moves (§7) but the field shape does not change.

## 3. Tool surface

> **This section is the load-bearing part of the spec for agent behavior.**
> The *prose of each tool description* is the artifact the agent actually
> reads. If you change these strings, you are changing the product. If the
> code disagrees with this section, the code is wrong; open an amendment
> (§13) rather than quietly editing prose to match what got shipped.

Two tools are added: `read` and `install_cookies`. `install_passkey` is
preserved with a new signature, and migrates from JSON to YAML to match
`install_cookies`'s file format. `read_profile` is deleted. There is
deliberately no list or search tool — see §4.2 for the immutability
invariant that makes them unnecessary.

### 3.1 `read`

**Tool name:** `read`
**Parameters:** `path: string` (required)
**Description — authoritative prose:**

```
Read a file from the Context list. The `path` argument is a name from
the tree shown in the Context section of the system prompt — that tree
is the full map of what's available. Returns the file's contents
verbatim as text. Binary files are not supported; attempts to read
binary content return an error. This is the tool to use when a story
names a user and you need their credentials, character notes, or any
other file the story references.
```

**Amendment note (PRI-1614).** The earlier prose named the on-disk
directory (`.gauntlet/context/`). That was a leak: the agent's mental
model should be "I have a list of files I can read," not "I have a
filesystem layout." The new prose describes paths as names from the
tree listing — the listing is the agent's only handle on this content,
not a directory it can navigate.

**JSONSchema:**

```json
{
  "name": "read",
  "description": "<above prose>",
  "parameters": {
    "type": "object",
    "properties": {
      "path": {
        "type": "string",
        "description": "A name from the Context tree (e.g. 'alice/credentials.md'). Must not contain '..' segments or start with '/'."
      }
    },
    "required": ["path"]
  }
}
```

**Behavior:**
- Resolves `path` under the configured context root
  (`<dataDir>/.gauntlet/context/`). Rejects `..` segments, absolute paths,
  and any path whose resolved absolute form does not live under the root.
  Matches the existing defence in `readProfile`.
- Reads the file as UTF-8. On a UTF-8 decode error, returns an error text
  result — never raises.
- On success, returns `{ text: <file contents> }`.
- On any error (not found, not a file, path escape, decode failure),
  returns `{ text: "Error: <message>" }`. The agent sees the error as a
  tool result; the run does not abort.

**Registration:** registered unconditionally when `.gauntlet/context/`
exists and is non-empty. If the directory is absent or empty, the tool is
not registered (matches v1's "invisible when unused" shape).

### 3.2 `install_passkey`

Preserved from v1 in function; re-parameterized to take an explicit file
path instead of a profile "name". Same CDP plumbing. Same Chrome gotcha
defences.

**Tool name:** `install_passkey`
**Parameters:** `path: string` (required)
**Description — authoritative prose:**

```
Install a passkey credential into the browser's virtual authenticator,
reading the credential YAML from a file under the project's context
directory. The path is relative to .gauntlet/context/ (example:
"alice/passkey.yaml"). You must re-call this tool after every navigate()
and before any click that triggers WebAuthn — Chrome clears virtual
authenticators on every same-target navigation, and the authenticator does
not survive. Calls are safe and cheap to repeat. The tool returns a
success message naming the rpId on success; on failure it returns an
error identifying the CDP step that failed.
```

**JSONSchema:**

```json
{
  "name": "install_passkey",
  "description": "<above prose>",
  "parameters": {
    "type": "object",
    "properties": {
      "path": {
        "type": "string",
        "description": "Path to the passkey YAML file, relative to .gauntlet/context/. Example: 'alice/passkey.yaml'."
      }
    },
    "required": ["path"]
  }
}
```

**Behavior:**
- Path resolution uses the same guard as `read`.
- Parses the file as YAML, validates required fields
  (`credentialId`, `rpId`, `privateKey`, `signCount`) and normalizes
  base64url byte fields to standard base64 (Ponder's
  `toStandardBase64` logic moves here — see §7).
- All CDP WebAuthn plumbing is unchanged from v1: pinned WebSocket via
  `chrome.webAuthnOpenSession`, full rebuild on every call, sanitized
  action-log context, teardown on adapter close.
- Registration: available only in the web adapter (unchanged from v1 — no
  passkey support in CLI/TUI).

> **Prose-break note.** The wording "after every navigate() and before any
> click that triggers WebAuthn" is load-bearing — Ponder's notes record
> that the agent learned the ordering *from this sentence alone*. Do not
> soften or abbreviate it.

### 3.3 Deletions

- `read_profile` — deleted. Its description, schema, and registration are
  removed.
- Any typed "profile catalog" / `Profile` / `KeychainEntry` type that
  might exist or get reintroduced — forbidden. The data store is
  untyped.

### 3.4 `install_cookies`

A new tool added under Plan E (description first, amendment-protocoled
here, then implemented). Companion to `install_passkey` for cookie-based
authentication, but distinct in lifecycle: cookies are browser state,
not session-scoped CDP state, so the tool has no teardown and does not
need to be re-called after navigation.

**Tool name:** `install_cookies`
**Parameters:** `path: string` (required)
**Description — authoritative prose:**

```
Install cookies into the browser, reading them from a YAML file under
the project's context directory. The path is relative to
.gauntlet/context/ (example: "alice/cookies.yaml"). The file is a list
of cookie entries; each entry mirrors Chrome's Network.setCookie
parameters (name, value, plus either url or domain+path, and optional
secure, httpOnly, sameSite, expires). Call this once, before navigating
to a cookie-gated origin. Cookies persist across same-origin
navigations; you do not need to re-call this tool. Note that the
browser performs an initial navigate before any tool runs, so for apps
that require a session cookie you must navigate again after installing.
The tool returns a per-cookie summary: how many were accepted, and
which entries (if any) Chrome rejected and why.
```

**JSONSchema:**

```json
{
  "name": "install_cookies",
  "description": "<above prose>",
  "parameters": {
    "type": "object",
    "properties": {
      "path": {
        "type": "string",
        "description": "Path to the cookies YAML file, relative to .gauntlet/context/. Example: 'alice/cookies.yaml'."
      }
    },
    "required": ["path"]
  }
}
```

**File format.** The YAML file is a list. Each entry is a Chrome
`CookieParam` object (see CDP `Network.setCookie`). Required: `name`,
`value`, and origin info — either `url`, or `domain` (with optional
`path`). Optional: `secure`, `httpOnly`, `sameSite`
(`Strict`/`Lax`/`None`), `expires` (Unix epoch seconds), `priority`,
`sameParty`, `sourceScheme`, `sourcePort`. Unknown fields are rejected
to catch typos like `samesite` vs `sameSite`. The parser passes
through verbatim — the spec author owns cookie semantics, not the
runner.

**Behavior:**
- Path resolution uses the same guard as `read` and `install_passkey`.
- Parses the file as YAML, validates each entry has `name`, `value`,
  and origin info. On parse error, surfaces YAML line/column.
- For each entry, issues `Network.setCookie` (singular, not the
  plural `setCookies` — the singular form returns per-cookie success).
  Aggregates per-cookie results: returns
  `{ accepted: [...], rejected: [{ name, reason }] }` to the agent.
  Partial success is normal — the agent learns which cookies got in.
- Sanitized action log: `install_cookies_ok` / `install_cookies_failed`.
  Logs `valueLength` only, never the cookie value bytes. Same secret
  hygiene as `install_passkey`.
- Registration: web adapter only. Available whenever
  `.gauntlet/context/` is non-empty (matches `read` and
  `install_passkey`).

> **Prose-break note.** The clauses "Call this once, before navigating
> to a cookie-gated origin", "Cookies persist across same-origin
> navigations", and "you must navigate again after installing" are
> load-bearing. Without these three sentences the agent does not
> understand the lifecycle difference from passkeys (which clear on
> navigate). Do not soften.

## 4. System prompt construction

The agent's system prompt gains one new section — **Context** — appended
after the "Reporting" section in `buildSystemPrompt`. This is the *entire*
interface through which the agent learns that `.gauntlet/context/` exists.
There is no Keychain, no profile list, no alias table.

### 4.1 The prose

When the source context directory is non-empty, the system prompt
appends exactly this block (interpolated where marked):

```
## Context

Below is the complete list of files available for this run. Use the
`read` tool with a name from this tree to fetch any file's contents.

Stories will often refer to users by name ("Alice", "as bob") without
spelling out credentials. When that happens, look for a matching path in
the tree below, `read` the relevant files, and use what you find to log
in via the regular browser tools. A profile directory typically contains
an identity file (prose describing the person) and a credentials file;
some also contain `passkey.yaml` for WebAuthn sign-in via
`install_passkey`.

This listing is the full map: it is built once at the start of the run
and does not change while the run is in flight, so you do not need to —
and cannot — re-list it. Every file you might need is in this tree; if a
path is not shown here, it does not exist.

{{TREE_LISTING}}
```

When the source context directory is absent or empty, the whole Context
section is omitted. No mention of context, no tool descriptions referring
to it — the agent should be invisible to context the same way v1's
keychain is invisible when no profiles exist.

> **Amendment note (PRI-1614).** Earlier prose named the on-disk
> directory `.gauntlet/context/` in three places (opening paragraph,
> closing paragraph, and the tree's subheading). That path was an
> implementation detail and should never have appeared in the agent's
> system prompt. The agent's mental model is a list of files it can
> read via the `read` tool — not a filesystem layout it can navigate.
> The new prose drops the leak; the section is still called "Context"
> because that name is a useful label for the agent's reference, not
> because the agent needs to know the storage path.

> **Prose-break note.** The framing — "list of files available", "stories
> refer to users by name", "tree is below" — is the spec. Every noun is
> intentional: "stories refer to users by name" cues the
> credential-discovery mental model; the tree is *ground truth* rather
> than a hint. The closing paragraph's "built once at the start of the
> run and does not change" is the prose face of the immutability
> invariant (§4.2) and is equally load-bearing.

### 4.2 Immutability invariant

**`.gauntlet/context/` is treated as immutable for the lifetime of a
run.** The tree listing is built once at turn 0 and is authoritative for
the whole run. No refresh mechanism is provided: no list tool, no search
tool, no re-read of the directory after the system prompt is assembled.
The agent holds the complete map before it takes its first action and
that map does not change.

This is a deliberate design statement, not an empirical defer. Three
things follow from it:

1. **Authors must not edit context files while a run is in flight.** If
   they do, the agent will act on stale information — the tree still
   shows the old entries, and `read` will return whatever the file
   contains at the moment of the call, which may disagree with the tree
   in size or in content. The invariant places the responsibility on the
   author: treat the directory as frozen once `gauntlet run` starts.
2. **The runner treats the directory as read-only.** Nothing in the
   Gauntlet code path writes into `.gauntlet/context/`. Concurrent runs
   may share a single context directory safely as long as no external
   process edits it.
3. **No `ls` / `grep` / refresh tool will be added.** The absence of
   those tools is intentional: if one were added, the invariant would
   become ambiguous ("what does the map mean?"), and the agent would
   gain a discovery surface it does not need. The tree *is* the
   discovery surface, and it is complete.

If a future use case genuinely requires mid-run context mutation, the
correct response is a spec amendment (§13) that reframes the invariant,
not a quiet addition of a list or refresh tool.

### 4.3 Tree format

`{{TREE_LISTING}}` is rendered by a compact printer with these rules:

1. Depth-first, alphabetical (case-insensitive). Directories first.
2. Each line: two spaces of indent per depth level, then the entry name,
   then — for files only — `  (N bytes)`.
3. Directories get a trailing `/`.
4. Hidden entries (name starting with `.`) are skipped.
5. **Budget:** 64 KB total and 400 entries total, whichever comes first.
   If either is exceeded, truncate depth-first and append a final line:
   `... (truncated: N more entries not shown — this run cannot see them)`.
   Truncation is a hard limit on agent visibility: there is no tool to
   retrieve the hidden entries mid-run. If truncation fires in practice,
   that is a signal to widen the budget (§12) or to reshape the context
   directory, not to add a discovery tool.

Example rendering for the anchor layout:

```
  alice/
    credentials.md  (52 bytes)
    identity.md  (188 bytes)
    passkey.yaml  (312 bytes)
  bob/
    credentials.md  (52 bytes)
    identity.md  (74 bytes)
```

Why byte counts: they cheaply signal which files are prose vs. structured
vs. large-fixture. A 312-byte passkey.yaml looks structured at a glance; a
188-byte identity.md looks like prose. No heuristic — the agent makes that
judgement itself.

### 4.4 Injection site

The prompt is assembled by `buildSystemPrompt(card, context?)` where
`context?` is the optional `{ tree: string; root: string }` produced by
the runner. The Context section is appended **after** the "Reporting"
section (i.e., it is the last block). This placement keeps it adjacent to
the tool results the agent will be looking at as it works. Full section
ordering:

1. Role + responsibility (unchanged)
2. Story Card (unchanged)
3. Acceptance Criteria (unchanged)
4. Reporting (unchanged)
5. **Context** (new — only when populated)

The tree is built **once per run, at turn 0**, baked into the system
prompt, and never rebuilt. Per §4.2 this is authoritative for the whole
run.

## 5. Story runner behavior

### 5.1 Browser state reset — one Chrome profile per story

**Decision:** every story run uses a unique, disposable Chrome
`--user-data-dir`. No state — cookies, localStorage, service workers,
IndexedDB, virtual authenticators (those are per-target anyway) — carries
from one story to the next.

**Mechanism.** `chrome-ws-lib.js` currently supports a named profile
(`chromeProfileName`, default `'superpowers-chrome'`) which maps to a
persistent directory under
`~/.cache/superpowers/browser-profiles/<name>/`. v1.5 introduces a
per-story profile name generated by the runner and handed in at Chrome
startup:

- **Local Chrome (`remote === false`).** The runner computes
  `profileName = \`gauntlet-run-${runId}-${storyId}\`` and passes it
  through `WebAdapterOptions.chromeProfileName` → `chrome.startChrome(headless, profileName)`.
  `startChrome` already accepts `profileName` (see `chrome-ws-lib.js`
  around line 1254) and already writes the profile dir under
  `~/.cache/superpowers/browser-profiles/<name>/`. On
  `WebAdapter.close()`, the runner *recursively deletes the profile dir*
  after `killChrome`. The deletion is best-effort (swallow errors,
  log an action-log entry `chrome_profile_cleanup_failed` on failure).

- **Remote Chrome (`remote === true`, e.g. the compose `chrome` service).**
  The runner cannot control the remote's data dir. Instead, on
  `WebAdapter.start`, it invokes `chrome.clearBrowserData(0)` (a new
  helper that fans out CDP `Storage.clearDataForOrigin` + `Network.clearBrowserCookies`
  + `Network.clearBrowserCache` + `ServiceWorker.stopAllWorkers` against
  the current target). This is a best-effort reset for the remote case.
  Documented explicitly as weaker than the per-profile-dir reset — the
  remote case is mostly for CI; CI should still isolate containers if
  strict isolation matters.

**Why one-profile-per-story rather than one-profile-per-fanout.** Stories
are the unit of isolation in Gauntlet's mental model. Fanout generates
*peer* stories that will be run against fresh state, not chained ones.

**Why Chrome profile dir, not `clearBrowserData`-only for local.** The
profile dir is the true source of truth; clearing by CDP is a heuristic
with known gaps (extensions, service worker registration metadata,
origin trial tokens). The profile-dir approach also isolates runs that
happen to overlap in time. Cheap, total, correct.

### 5.2 Concurrency by default

**Decision:** story runs do not block or queue on each other. They run
in parallel from the runner's point of view. Each run holds its own
Chrome profile dir, its own `EvidenceLogger`, its own agent message
history, and its own context-tree snapshot. Nothing is shared.

In v1.5 the CLI still launches one run at a time (single-card
invocation), but nothing assumes sequentiality: the server's
`/api/run/:id` route already handles concurrent POSTs, and the fanout
path can be driven to dispatch in parallel. The change in v1.5 is
normative — **story ordering is explicitly forbidden as a design assumption**:

- Cards must not depend on side effects of other cards.
- The `parent:` frontmatter field remains (fanout uses it for lineage)
  but does not imply ordering. It means "this card was generated from
  that one", not "run that one first".
- The runner will *not* grow any inter-story sequencing primitive — no
  `depends_on`, no ordered batch, no "group" abstraction.

### 5.3 "As X" identity convention

**Decision:** story cards express identity in prose using the phrase
**"as X"** (or equivalent — "Alice publishes", "Bob, who is not on her
friends list"). No frontmatter field for active identity, no
adapter-side "current user" primitive.

The agent, at turn 0, receives the card, the Context section with the
tree, and no other hint. When the card's narrative reaches a point
requiring identity X:

1. The agent finds X in the tree (e.g., `alice/` directory or `alice.md` file).
2. It reads what it needs — typically `alice/identity.md` for character
   and `alice/credentials.md` for login fields, or both in one file.
3. It uses existing browser tools (`navigate`, `type`, `click`) to log
   in.
4. For passkey-based sites, it calls `install_passkey({ path: "alice/passkey.yaml" })`
   after navigating to the login page and before clicking the passkey
   button.

**Logging out between identities.** The same mechanism: the agent sees
the narrative shift ("Bob then tries to find it"), logs out via the site
UI if the site needs it, then reads Bob's files. Because browser state
resets *between runs* but **not within a run**, the agent is responsible
for logging out within a single run that switches identities.

**No pre-auth stage.** The runner does not log in as anyone before the
agent starts. A run starts with the target URL's default page (logged
out) and whatever state the fresh Chrome profile provides (nothing).

### 5.4 Concurrency model details

- Each story run gets its own `EvidenceLogger`, writing to its own output
  directory. No cross-run state.
- Chrome profile directories (local case) live under
  `~/.cache/superpowers/browser-profiles/gauntlet-run-<runId>-<storyId>/`
  and are deleted on `close()`. Stale directories from crashed runs are
  left alone — a small maintenance task could add a sweeper later, but
  that's not in v1.5 scope.
- In the remote-Chrome case, the runner ensures a single CDP target per
  run by creating a fresh tab for each run instead of reusing tab 0.
  Tab 0 remains the implicit "first tab" the adapter operates against;
  in remote mode the adapter creates its own tab on `start()` and drives
  it via its own index. **Note:** this interacts with the
  tab-0-hardcoded assumption in `passkey-tool.ts` — see §7 for how the
  rewiring handles this.

## 6. Card conventions

A v1.5 card is a v1 story card with these clarifications. **No schema
changes.** Cards from v1 continue to parse.

### 6.1 Fields

The existing `StoryCard` shape (see `src/format/story-card.ts`) is
unchanged. The `parent:` field becomes explicit in its meaning
(lineage, not ordering). No new frontmatter fields in v1.5.

### 6.2 Expressing identity in prose

Authors name identities in the card body or acceptance criteria using
any natural phrasing — "as alice", "Alice publishes", "acting as bob",
"Bob Wilson". The agent matches against the `.gauntlet/context/` tree
using its own reading comprehension. The author's job is to make the
match obvious; a tree entry named `alice/` plus a card that says
"Alice" will connect.

### 6.3 Anchor example — Alice-and-Bob privacy

```markdown
---
id: privacy-001
title: Alice's posts only visible to friends
status: ready
---

Alice publishes a post with "friends only" visibility. Bob, who is not
on her friends list, then tries to find it via search and in the feed.

## Acceptance Criteria
- Privacy level is clearly indicated at post time
- The post's distribution is clearly indicated after posting
- The post doesn't appear in search for bob
- The post doesn't appear in bob's feed
```

With `.gauntlet/context/alice/` and `.gauntlet/context/bob/` as in §2.4,
this is a complete v1.5 story. The narrative carries the ordering ("Alice
publishes... Bob then tries..."); the agent reads the context on demand;
the browser state is fresh at the start of the run and isolated from
every other run.

### 6.4 Procedural alternative (when determinism matters)

Authors can still spell out steps explicitly when the narrative style is
not precise enough. The conclave explicitly preserves this style for
critical regressions:

```markdown
---
id: privacy-001
title: Alice's posts only visible to friends
status: ready
---

Verify that a friends-only post is not visible to non-friends.

## What to Test

1. Read alice's identity and credentials from `.gauntlet/context/alice/`.
2. Navigate to the login page. If the site uses passkey sign-in, call
   `install_passkey({ path: "alice/passkey.yaml" })` before clicking
   the passkey button.
3. Log in as Alice.
4. Publish a short post with privacy level set to "friends only".
5. Note the post URL.
6. Log out.
7. Read bob's credentials. Log in as Bob.
8. Visit the URL from step 5. Confirm you cannot see the post.
9. Check the feed and search for a distinctive word from the post.
   Confirm neither shows it.

## Acceptance Criteria
- The post doesn't appear in search for bob
- The post doesn't appear in bob's feed
```

Both styles are correct. Both run against the same v1.5 runner.

## 7. Web adapter changes and the `src/context/` module

### 7.1 Decoupling `format/profile.ts` — per-file lifecycle

`src/format/profile.ts` currently holds `listProfiles` / `readProfile`
(profile file access), `listPasskeyProfiles` / `readPasskey` (passkey
JSON access), the `PasskeyCredential` interface, and `toStandardBase64`
(the Chrome CDP encoding fix). None of these survive v1.5 unchanged. The
v1.5 layout introduces a **new top-level module** — `src/context/` —
whose charter is "the semantics of `.gauntlet/context/`, agnostic of
transport." Generic filesystem primitives live there. Passkey-specific
code moves next to the web adapter that owns it. Per-file lifecycle:

- **`src/format/profile.ts`** — *deleted in Phase 3 (WP3.1).* No
  successor. The first-class typed profile concept does not survive;
  story cards (`src/format/story-card.ts`) are unaffected.
- **`src/adapters/profile-tool.ts`** — *deleted in Phase 3 (WP3.1).*
  The `read_profile` tool is superseded by the generic `read` primitive
  in `src/context/read-tool.ts`.
- **`src/context/read-tool.ts`** — **new (Phase 1, WP1.7).** Houses
  `buildReadTool(contextRoot): ReadTool | null`. A pure read-only
  filesystem primitive. No knowledge of profiles, passkeys, or
  credentials. Returns `null` when `contextRoot` does not exist or is
  empty, mirroring v1's "invisible when unused" shape.
- **`src/context/tree.ts`** — **new (Phase 1, WP1.7).** Houses
  `renderContextTree(contextRoot, options?): string` — the compact tree
  renderer whose output is spliced into the system prompt (§4.3). The
  runner calls it once per run and passes the resulting string into
  `runAgent`.
- **`src/context/path.ts`** — **new (Phase 1, WP1.7).** Houses
  `resolveInside(root, rel): string`, the shared path guard used by
  both `read` (in `src/context/read-tool.ts`) and `install_passkey`
  (in `src/adapters/web/passkey.ts`). Its behavior matches §3.1
  verbatim: reject `..` segments in the input, reject absolute paths,
  resolve the path against `root`, and reject results that do not
  live under `root`. Returns the resolved absolute path on success;
  throws on failure. This is stricter than v1's per-file guards in
  `format/profile.ts`, which rejected only `/` and `\` character
  literals and missed several edge cases.
- **`src/adapters/web/passkey.ts`** — **new (Phase 1, WP1.5).**
  Consolidates everything passkey-specific previously split between
  `src/adapters/passkey-tool.ts` and `src/format/profile.ts`:
  - `PasskeyCredential` interface (moved verbatim).
  - `toStandardBase64` (moved verbatim, internal). The Chrome 147
    base64url-vs-base64 bug comment travels with the function — the
    knowledge and the code must not be separated.
  - `readPasskeyFile(absolutePath): PasskeyCredential` — takes an
    already-resolved absolute path. The tool does relative-path
    resolution via `resolveInside` from `src/context/path.ts` before
    calling in. No `dir`/`name` pair anymore.
  - `WebAuthnSession`, `WebAuthnDriver`, `VirtualAuthenticatorOptions`
    interfaces and `DEFAULT_AUTHENTICATOR_OPTIONS` (moved verbatim).
  - `credentialContext` helper for sanitized action-log entries
    (moved verbatim).
  - `buildInstallPasskeyTool(contextRoot, tab, driver, logger)` — same
    shape as v1's tool, with `contextRoot` replacing `profilesDir` and
    the execute path using the explicit `path` parameter.
- **`src/adapters/passkey-tool.ts`** — *deleted in Phase 1 (WP1.5).*
  Its contents move verbatim to `src/adapters/web/passkey.ts`. The
  move is a relocation, not a rewrite: the CDP ceremony, the
  sanitized action-log context, and the pinned-WebSocket requirement
  are all preserved unchanged. No deprecation window, no shim.
  Passkey is a web-adapter-only concept; there is no reason for it
  to live under the generic `adapters/` directory.

**Why `src/context/` rather than extending `src/adapters/`.** The
`read` tool and the tree renderer are pure filesystem primitives —
adapter-agnostic in the strongest sense. `src/adapters/` is reserved
for code that implements or wires into the `Adapter` interface, which
is transport-specific. Placing `read` under `src/adapters/` would
repeat the category error that landed `PasskeyCredential` in
`src/format/`: putting generic code next to its callers instead of
in the layer that describes what it *is*. Naming the new module
after the v1.5 concept ("context") keeps the directory tree legible
to the next Bob who walks in — "the semantics of `.gauntlet/context/`
live here" is a sentence the layout itself makes obvious.

### 7.2 `install_passkey` with an explicit path

The tool executes as follows:

1. Parse `path` from args. Reject empty, absolute, and `..`-containing
   paths with an action-log-visible error.
2. `resolved = resolvePath(contextRoot, path)`. If
   `!resolved.startsWith(contextRoot + "/")`, reject with
   "path escapes context root".
3. `credential = readPasskeyFile(resolved)`. Error surfaces as a tool
   result.
4. All CDP steps as in v1 — open session, add virtual authenticator, add
   credential, success message. Failure mode, sanitized context logging,
   pinned-WebSocket requirement all unchanged.

### 7.3 WebAdapter constructor

```ts
export interface WebAdapterOptions {
  chrome?: ChromeEndpoint;
  /**
   * Path to the context root (.gauntlet/context/). When set and the
   * directory is non-empty, the adapter registers `read` (and
   * `install_passkey` for any readable passkey.yaml paths) and produces
   * a tree snapshot for the system prompt. When absent or empty,
   * context tools are not registered.
   */
  contextRoot?: string;
  logger?: EvidenceLogger;
  /**
   * Used for browser state isolation. The runner generates this per
   * run; WebAdapter passes it to chrome.startChrome() as the profile
   * name so each run has its own Chrome --user-data-dir.
   */
  chromeProfileName?: string;
}
```

- `profilesDir` is **removed**.
- `buildReadProfileTool` import is **removed**.
- `contextRoot` replaces `profilesDir` everywhere it wired into
  profile/passkey tools.
- `chromeProfileName` is new (§5.1).

### 7.4 CLI and TUI adapters

- `src/adapters/cli/adapter.ts` and `src/adapters/tui/adapter.ts`: remove
  the `profileTool` field and `profilesDir` option. Replace with a
  `contextRoot` option that wires `buildReadTool` (from
  `src/context/read-tool.ts`) the same way `WebAdapter` does. `read`
  works identically in CLI/TUI contexts — it is a filesystem tool, not
  a web tool. `install_passkey` remains web-only.

### 7.5 Exposing the tree to the runner

`WebAdapter` needs to give the runner the rendered tree for
`buildSystemPrompt`. Two options:

1. Runner builds the tree itself (calls `renderContextTree(contextRoot)`
   from `src/context/tree.ts`), then passes the result to both the
   adapter (via `WebAdapterOptions.contextTree`) and `buildSystemPrompt`.
   The adapter only needs the root path for its tools.
2. Adapter builds the tree and exposes it via a method.

**Pick option 1** — the tree is a context-module concern, not an
adapter concern. `src/context/tree.ts` is imported directly by the
runner, and all three adapters share it without each one
reimplementing.

### 7.6 Runner wiring (`src/cli/run.ts`, `src/api/routes/run.ts`)

```ts
const contextRoot = join(config.dataDir, ".gauntlet", "context");
const contextTree = renderContextTree(contextRoot);  // may return "" if empty/absent
const runId = makeRunId();  // existing helper, or new short-id util
const chromeProfileName = `gauntlet-run-${runId}-${card.id}`;

// ... construct adapter ...
adapter = new WebAdapter({
  chrome: chromeOpt,
  contextRoot,
  chromeProfileName,
  logger,
});
await adapter.start(target);

const result = await runAgent(card, adapter, client, logger, target, {
  contextTree,
  contextRoot,
});
```

`runAgent` grows an optional 6th argument (or an options slot) for the
context data; it forwards to `buildSystemPrompt`.

## 8. Observer session — untouched

The observer session (`chrome.openObserverSession`, the
`BrowserEventCategory` streams, `logger.logBrowserEvent`, the per-category
`.jsonl` files) is **preserved exactly as Ponder built it.** No new
events, no renamed categories, no moved files, no altered error
handling.

**Why it is untouched:** Ponder's implementation notes record that the
observer is what made the CDP WebAuthn gotchas diagnosable at all. It is
bug-report-grade infrastructure paid for under fire. Nothing in v1.5
changes how the agent observes the browser — only how it discovers
context and how stories relate to each other. Touching the observer
would be pure risk.

**Plan consequence:** WP1.6 (`Observer session verification`) in the
implementation plan is explicitly a *regression check*, not a change.
The Phase 1 work verifies the observer still streams the expected event
categories after the rest of the mechanical changes land, then nothing
more.

**Forbidden edits.** The following files are off-limits in v1.5:

- `src/adapters/web/lib/chrome-ws-lib.js` — the `openObserverSession`
  function, the CDP domain enables (`Runtime.enable`, `Log.enable`,
  `Network.enable`), and the event forwarding.
- `src/evidence/logger.ts` — `BrowserEventCategory`, `logBrowserEvent`,
  per-category JSONL writer.
- `src/adapters/web/adapter.ts` — the observer session opening in
  `start()` and closing in `close()`.

*Exception:* the observer-touching code in `WebAdapter.start` lives
alongside the `chrome.startChrome` and `chrome.navigate` calls that
Phase 1 needs to modify for browser state reset (§5.1). Those
modifications are allowed; the observer block itself (after the
navigate) must stay bit-identical.

## 9. Model defaults — Sonnet 4.6 floor

**Decision:** the v1.5 default agent model is Sonnet 4.6. Runs below
that floor are rejected at config load time with a clear error.

### 9.1 The floor

Define a new constant in `src/config.ts`:

```ts
// Models at or above the floor are considered "v1.5-capable".
// This list is additive as newer models ship; anything not here is
// rejected by requireLlmCapable and mergeRunConfig (loadConfig stays
// introspectable for `gauntlet config`).
const AGENT_MODEL_FLOOR_ALLOWLIST: ReadonlySet<string> = new Set<string>([
  "claude-sonnet-4-6",
  "claude-opus-4-6",
  // Add dated variants here when they ship; do not invent them.
  // Explicitly NOT: claude-sonnet-4, claude-sonnet-4-0, claude-sonnet-4-20250514.
]);
const DEFAULT_AGENT_MODEL = "claude-sonnet-4-6";
```

The default string `claude-sonnet-4-6` was already correct at
`src/config.ts:104` before WP1.4 started (§12.1 confirms). WP1.4's work
was adding the allowlist + enforcement sites and scrubbing stale
`claude-sonnet-4-20250514` references from `.env.example` and
`README.md`.

### 9.2 Enforcement

The floor check runs in two places, both on the "about to actually run"
path. `loadConfig` is deliberately **exempt** — it must remain
introspectable by `gauntlet config` on broken environments, so it
never throws for a below-floor model. Instead:

- **`requireLlmCapable(config)`**, after the existing provider-presence
  check, calls `checkAgentModelFloor(config.models.agent, "GAUNTLET_AGENT_MODEL")`.
  Both CLI `run` and server startup already call this gate.
- **`mergeRunConfig(app, body)`**, after computing the effective model,
  calls `checkAgentModelFloor(effective.model, label)` where `label` is
  `"run.model"` when the body supplies a model and `"GAUNTLET_AGENT_MODEL"`
  when falling back to the server default. This labels the error with
  the knob the user actually set. The route layer converts the thrown
  error to a `400`.

On miss, the error message is:

```
<label>: model "<name>" is below the Gauntlet v1.5 floor. The floor is
Sonnet 4.6; use one of: claude-sonnet-4-6, claude-opus-4-6.
```

The two-site split (not just `requireLlmCapable`) covers the route-layer
per-request override seam: a below-floor `model` in the POST body would
otherwise slip past the startup gate.

`.env.example`, `README.md`, `docs/primer.md`, `src/cli/args.ts`'s usage
string, `ui/src/components/NewRunModal.tsx`'s placeholder — all updated
to Sonnet 4.6. Documented in the plan (WP1.4 verification step).

## 11. Security notes

The threat model is unchanged from v1 — credentials still appear in the
message stream once the agent reads them, still land in LLM provider
logs, still end up in `EvidenceLogger` action logs, and may still leak
into screenshots depending on how the app under test renders password
fields. The v1 `docs/profiles-spec.md` §"Security notes" text applies
verbatim; the specifics are re-stated here so this spec is
self-contained.

- **Context files live on local disk.** Holders of the disk hold the
  credentials. Gauntlet provides no encryption, no secret vault, no
  interaction with system credential managers.
- **Credentials enter the message stream on first `read`.** Every
  subsequent turn that includes the message stream (which is all of
  them) sends them to the LLM provider again. Anthropic / OpenAI
  retention policies apply.
- **Action log.** Every `read` tool call is recorded with its arguments
  (the file path), and the tool result text (the file contents) is part
  of the agent-visible transcript and the LLM provider's visible
  context. Downstream `type` tool calls log their `text` argument — the
  typed password — verbatim.
- **Screenshots.** Most apps mask password fields. Some briefly show
  plaintext. Most apps do not mask credential inputs in UI labels
  ("Logged in as alice@example.com") which will appear in every
  screenshot after sign-in.
- **Passkey JSON.** `install_passkey` never includes the private key in
  its tool result text or in action-log context — only sanitized
  metadata (rpId, signCount, field lengths). The raw key still lives on
  disk in the context file. The tool's pinned CDP session is scoped to
  the run and torn down on adapter close; no OS keychain is touched.

**Baseline rule, unchanged:** use throwaway test credentials on dev /
staging databases. Never production accounts. Never a password reused
anywhere else.

**Gitignore.** `.gauntlet/context/` should be gitignored by default in
any project using Gauntlet v1.5. The runner will not auto-create a
`.gitignore` in the directory (v1 deferred this; v1.5 continues to
defer it — it's a one-line README recommendation, not code).

**Lazy-loading carryover.** The v1 lazy-loading shape is preserved by
this design: a file's contents only enter the message stream when the
agent `read`s it. Files the run never touches never reach the LLM. The
tree listing at turn 0 contains names and byte counts only — no
contents — so a populated `.gauntlet/context/` does not bleed secrets
into the system prompt.

the tool's reach narrow — and revisit only if Phase 2 surfaces a

## 13. Amendment protocol (the prose-break rule)

This spec's prose — tool descriptions, system-prompt sections, the
Context section framing, card conventions — is **authoritative**. If
implementation reality contradicts what the spec says (e.g., the agent
reliably misreads "Context" as something else, or the tree format turns
out to produce broken Markdown in the system prompt, or Chrome
introduces a fourth WebAuthn gotcha), the correct response is to open
an amendment rather than quietly change the prose to match the code.

**Amendment process:**

1. The Bob hitting the contradiction writes a short amendment: "Section
   X says Y. Reality says Z. Here's the minimum prose change." Include
   evidence (log excerpt, repro).
2. Matt reviews and approves or rejects.
3. On approval, the spec is updated in a single commit that changes
   only the prose and references the amendment. Code changes follow
   afterward.

This is Cyrano's rule from the conclave, formalized. It is not a
bureaucratic hurdle — it takes minutes — but it preserves the property
that the spec's prose is the source of truth, and it keeps the agent's
environment under deliberate control.

## 14. What v1.5 is not

A short list to prevent scope creep:

- **Not a full rewrite.** v1 is the teacher. Refactor in place.
- **Not a new card schema.** Cards stay identical; the only change is
  how the runner *interprets* prose (ordering assumption removed,
  "as X" convention documented).
- **Not a new adapter interface.** `Adapter` gains no methods. Context
  tools wire through existing `toolDefinitions` / `executeTool`
  machinery. Same contract.
- **Not a credential vault or secret manager.** The threat model is
  unchanged.
- **Not multi-tab support for WebAuthn.** The tab-0 hardcoding in the
  passkey tool is a known limitation (Ponder's notes flag it); v1.5
  does not fix it.
- **Not an ordered story batch runner.** The runner does not grow one,
  and cards must not assume one.
- **Not a fanout semantics change.** Fanout continues to generate peer
  cards with `parent:` lineage. The lineage is informational.
- **No `grep` tool.** If Phase 2 observation surfaces a case where the
  agent would benefit from substring search across context files, add
  it under Plan E discipline — write the tool description first,
  amendment-protocol it into §3, then implement. Do not add it
  speculatively.

If you feel the urge to do any of the above, stop and open a
conversation with Matt.
